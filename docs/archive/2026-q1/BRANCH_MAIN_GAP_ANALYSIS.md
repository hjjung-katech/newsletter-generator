## 3) 변화의 비즈니스/기술 가치 평가

## 높은 가치

1. **품질 시스템의 내재화**: 테스트 + 로컬 CI + 문서가 함께 올라가 단발성 개선이 아닌 지속 가능성 확보.
2. **운영 안정성**: graceful shutdown, 스케줄/시간 동기화는 실제 장애/클레임 감소와 직결.
3. **배포 현실성 개선**: exe/경로 분리는 사용자 배포 채널 확장에 필수.

## 주의 필요

1. **변경 폭이 크고 분산됨**: 기능/인프라/문서가 한번에 섞여 회귀 위험 상승.
2. **중복/연속 커밋 흔적**: 유사 메시지의 연속 커밋(예: local CI 관련)이 있어 이력 가독성 저하.
3. **핵심 파일 집중 변경**: `web/app.py` 등 단일 파일에 책임 집중 가능성.

## 4) 가장 적절하고 합리적인 통합 원칙 (의사결정 기준)

- **원칙 1: 위험도 기반 순차 통합**
  - 사용자 기능보다 장애 리스크를 먼저 축소하는 순서로 병합.
  - 권장 순서: `CI/품질 기반` → `런타임/바이너리` → `스케줄러 신뢰성`.
- **원칙 2: 브랜치 분할 + 게이트 고정**
  - 한 번에 `work`를 합치지 않고 주제별 릴리즈 브랜치로 분할해 통제.
- **원칙 3: 테스트는 “선행 조건”**
  - 병합 전 테스트 통과를 결과가 아니라 진입 조건으로 관리.
- **원칙 4: 추적 가능한 릴리즈**
  - 태그/CHANGELOG를 병합 단위와 1:1로 맞춰, 문제 발생 시 역추적 가능하게 운영.

## 5) 앞으로 의미 있게 관리하는 방법 (권장 운영 모델)

### 5-1. 브랜치/릴리즈 관리

1. `main` 기준선 복구(또는 선언):
   - 현재 `work`를 즉시 합치기보다, 주제별로 릴리즈 브랜치를 분할:
     - `release/ci-platform`
     - `release/runtime-binary`
     - `release/scheduler-reliability`
2. 각 릴리즈 브랜치에서 회귀 테스트 및 스모크 테스트를 통과시킨 뒤 순차 병합.
3. 병합 단위별 태그(`vX.Y.Z`)와 CHANGELOG 섹션 분리로 추적성 확보.

### 5-2. 품질 게이트 운영

- PR 머지 조건을 최소 다음 4개로 고정:
  1) format/lint 통과
  2) 핵심 단위 테스트 통과
  3) 스케줄/종료 관련 통합 테스트 통과
  4) 보안 스캔 결과(신규 high 이슈 0)
- 로컬 pre-push 훅 실패 시 우회 금지(문서 예외 절차만 허용).

### 5-3. 회귀 방지 전략

- 장애 고위험 영역(스케줄러, 종료, 바이너리 런타임)을 “변경 시 필수 테스트” 영역으로 지정.
- 테스트 카테고리(quick/full/nightly)를 분리:
  - quick: 개발자 로컬, 5분 이내
  - full: PR 게이트
  - nightly: 장기 시나리오(시간대/스케줄 반복)

### 5-4. 운영/관측 지표(KPI)

- 배포 실패율(주간)
- CI 최초 통과율(First Pass Rate)
- 롤백 건수
- 스케줄 누락/중복 실행 건수
- 비정상 종료율 및 종료 처리 시간

### 5-5. 문서 거버넌스

- 코드 변경 PR에 “관련 문서 업데이트 체크박스”를 의무화.
- 아키텍처/운영 문서는 분기별 점검일 지정(오너 명시).

## 6) 구체 실행 계획: 테스트·관리·기능 통합

### 6-1. 통합 순서별 테스트 매트릭스

1. `release/ci-platform`
   - 목표: 품질 게이트 선행 확보
   - 필수 검증: lint/format, 핵심 unit, pre-push 훅 동작
   - 통과 기준: PR 게이트 100% 통과 + 신규 high 보안 이슈 0

2. `release/runtime-binary`
   - 목표: exe/path/runtime 안정화
   - 필수 검증: packaging 관련 테스트, 경로 분리 회귀, 시작/종료 smoke
   - 통과 기준: 배포 패키지 실행 성공 + 종료 처리 성공

3. `release/scheduler-reliability`
   - 목표: 스케줄 누락/중복/시간대 이슈 최소화
   - 필수 검증: schedule/shutdown 통합 테스트, 반복 실행 시나리오
   - 통과 기준: 누락/중복 0, 장기 시나리오에서 비정상 종료율 목표치 이내

### 6-2. 테스트 운영 절차(실행 주기)

- 매 커밋(로컬): `quick`
  - lint + 핵심 unit + 변경영역 필수 테스트
- 매 PR: `full`
  - quick + 통합 테스트 + 보안 스캔
- 매일 야간: `nightly`
  - 시간대/반복 스케줄/장시간 종료 복구 시나리오

### 6-3. 변경 통제 절차(운영)

- 위험 영역 파일(`web/app.py`, `web/schedule_runner.py`, `web/graceful_shutdown.py`, `newsletter/utils/shutdown_manager.py`) 변경 시:
  1) 코드 오너 1인 이상 승인
  2) 관련 통합 테스트 의무 첨부
  3) 릴리즈 노트에 영향 범위/롤백 방법 명시

### 6-4. 릴리즈·롤백 관리

- 릴리즈 노트 고정 섹션:
  - 사용자 영향
  - 운영 영향
  - 필수 모니터링 지표
  - 롤백 절차
- 배포 후 24시간 관측 윈도우 운영:
  - 스케줄 누락/중복, 비정상 종료율, 롤백 필요 여부를 집중 모니터링

## 8) PR 생성 방식과 main 통합 순서 (실행용 플레이북)

### 8-1. PR 생성 규칙(브랜치별 공통)

- PR은 아래 릴리즈 브랜치별로 **분리 생성**:
  1) `release/ci-platform` → `main`
  2) `release/runtime-binary` → `main`
  3) `release/scheduler-reliability` → `main`
- 각 PR은 “한 가지 목적(one theme)”만 포함하고, 범위를 넘는 변경은 다음 PR로 이월.
- PR 본문은 아래 6개 섹션을 고정 템플릿으로 사용:
  - Summary
  - Scope (in/out)
  - Risk
  - Test Evidence
  - Rollback Plan
  - Docs Updated

### 8-2. PR 생성 절차(체크리스트)

1. 기준선 동기화
   - `main` 최신 반영 후 릴리즈 브랜치 rebase/merge
2. 변경 범위 검증
   - 해당 릴리즈 목적 외 파일 변경 제외 확인
3. 테스트 실행
   - quick → full 순으로 통과 증빙 수집
4. PR 오픈
   - 라벨: `release`, `risk:*`, `area:*`
   - 리뷰어: 코드 오너 + 운영 담당 포함
5. 머지 조건 확인
   - 5-2 품질 게이트 4개 모두 통과
6. 머지 후 작업
   - 태그 발행, CHANGELOG 업데이트, 24시간 모니터링 시작

### 8-3. main 통합 순서(권장)

1. **1차:** `release/ci-platform`
   - 이유: 이후 PR들의 품질 게이트 기반을 먼저 안정화
2. **2차:** `release/runtime-binary`
   - 이유: 배포/실행 환경 리스크를 조기에 제거
3. **3차:** `release/scheduler-reliability`
   - 이유: 가장 운영 민감한 스케줄/종료 로직을 강화된 게이트 하에서 통합

### 8-4. 각 PR의 승인/머지 기준

- 승인 기준
  - 기본 승인 필수 조건 없음
  - high-risk 파일 변경 시 코드 오너/운영 담당 리뷰 요청 + 통합 테스트 증빙 필수
- 머지 방식
  - 기본: squash merge (히스토리 가독성)
  - 예외: 롤백 단위가 중요하면 merge commit 사용
- 머지 타이밍
  - 업무시간 초반(모니터링 가능 시간 확보) 우선

### 8-5. 통합 후 검증 및 실패 대응

- 머지 직후
  - smoke 실행, 핵심 API/스케줄 등록/종료 동작 확인
- 24시간 관측
  - KPI: CI First Pass Rate, 스케줄 누락/중복, 비정상 종료율
- 실패 시
  - 즉시 롤백(태그 기준), 원인 분류(테스트 누락/환경 차이/코드 결함), 재PR
